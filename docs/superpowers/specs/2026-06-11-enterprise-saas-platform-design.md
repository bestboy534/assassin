# 企业软件管理平台总体设计规格

**日期：** 2026-06-11  
**状态：** 已确认方向，待书面规格复核  
**产品形态：** 中文公开官网 + 多租户企业工作台 + AI 账单审计与退订 Copilot  
**目标读者：** 产品、设计、前端、后端、测试、安全、运维与实施团队

---

## 1. 文档目的

本规格把当前仓库中的两套能力统一为一个完整产品：

1. 现有中文企业软件管理官网负责获客、内容传播、演示预约和产品教育。
2. 现有账单解析后端成为企业工作台中的 AI 账单审计能力。
3. 新建可登录的企业工作台，承载应用目录、采购审批、合同续订、支出支付、会计自动化、集成、安全合规与分析报表。

本规格定义产品边界、功能要求、架构边界、数据模型、接口原则、权限体系、安全要求、测试策略和交付顺序。后续所有模块的开发计划必须引用本规格，不得自行缩小范围或改变关键业务定义。

## 2. 当前状态审计

### 2.1 已有公开站

当前 `frontend/src/App.tsx` 包含 45 个哈希路由和多种公开页面样式，覆盖：

- 首页
- 财务、采购、IT、运营团队方案
- 审批、优化、支付、应用目录、分析、会计、集成、安全、合规等解决方案
- 博客、报告、播客、媒体、招聘、隐私、条款等内容页
- 登录、定价、预约演示和开始使用入口

当前问题：

- 公开站与真实产品没有路由边界，所有页面集中在约 2,200 行的 `App.tsx`。
- 登录、注册、定价、预约演示等入口只有展示页面，没有可提交的真实流程。
- 内容写死在前端，缺少 CMS、SEO 元数据、站点地图和内容发布流程。
- 使用哈希路由，不利于公开页面 SEO、服务端部署和深链接。
- `index.css` 超过 2,200 行，缺少设计令牌和按域拆分。

### 2.2 已有账单审计 MVP

当前后端基于 FastAPI，已有：

- `POST /api/analyze`：解析账单文本。
- `GET /api/history`：查询分析历史。
- `GET /api/history/{run_id}`：查询单次分析结果。
- CSV、Apple、Stripe、PayPal、Google Play 来源类型。
- 规则解析与可选 LLM 解析。
- 币种归一化、月化成本、风险类型和退订路径匹配。
- SQLite 结构化结果存储，不保存原始账单。

当前问题：

- 没有用户、组织、成员、权限和会话。
- 分析历史不属于任何组织，无法做多租户隔离。
- 状态更新只存在前端 LocalStorage，没有后端持久化。
- 数据模型只覆盖一次性分析结果，没有应用、供应商、合同、交易、发票和续订等正式实体。
- SQLite 适合本地验证，不适合目标企业产品。
- 缺少任务队列、对象存储、病毒扫描、审计日志、幂等和重试机制。

### 2.3 已有但未接入的前端代码

`frontend/src/api.ts`、`store.ts`、`types.ts` 和 `components/` 下的账单审计组件仍存在，但未接入当前公开站。规划要求复用其中的业务语义，不直接把旧组件原样嵌入新工作台。

### 2.4 仓库卫生问题

当前存在浏览器调试目录 `frontend/.edge-codex` 被加入 Git 暂存区的情况。正式开发前必须：

- 从 Git 索引中移除该目录。
- 在 `.gitignore` 中忽略浏览器资料目录、构建产物、数据库文件、日志和测试截图。
- 保留用户已有业务改动，不通过重置或覆盖方式清理。

## 3. 产品愿景与成功标准

### 3.1 一句话定位

让企业在一个平台中发现、购买、支付、使用、续订和优化所有软件，并通过 AI 自动识别浪费与风险。

### 3.2 核心价值

- **财务团队：** 看清全部软件支出，减少浪费，缩短月末对账时间。
- **采购团队：** 将需求、预算、谈判、审批和合同纳入可追踪流程。
- **IT 与安全团队：** 识别影子 IT、权限风险、供应商风险和离职访问残留。
- **业务负责人：** 快速申请工具，了解审批状态，管理席位和续订决策。
- **管理层：** 用统一数据查看软件投资回报、预算执行和节省成果。

### 3.3 第一阶段商业成功指标

- 新组织首次价值时间不超过 30 分钟。
- 至少 70% 的试用组织成功导入一份账单或应用清单。
- 至少 50% 的活跃组织在首周发现一个可执行的节省机会。
- 采购申请从提交到首次审批的中位时间不超过 4 小时。
- 续订提醒覆盖率达到 95%。
- 高风险权限或影子应用发现后 7 天内处理率达到 70%。
- 付费组织月活跃率达到 65%。

### 3.4 产品质量指标

- 核心页面可用性不低于 99.9%。
- 常规 API P95 响应时间不超过 500ms。
- 大文件导入通过异步任务处理，提交接口 2 秒内返回任务 ID。
- 多租户越权测试必须为零缺陷。
- 关键写操作审计覆盖率为 100%。
- WCAG 2.2 AA 级可访问性作为发布门槛。

## 4. 用户与角色

### 4.1 平台级角色

| 角色 | 职责 |
|---|---|
| 平台超级管理员 | 管理租户、套餐、功能开关、系统任务、全局内容和支持升级 |
| 平台支持人员 | 在受控授权下查看组织诊断信息，不默认访问客户业务数据 |
| 内容编辑 | 管理博客、报告、播客、媒体、职位和 SEO 内容 |

### 4.2 组织级角色

| 角色 | 主要权限 |
|---|---|
| 组织所有者 | 组织设置、套餐、成员、SSO、数据导出和删除 |
| 组织管理员 | 成员、角色、集成、审批流程和安全配置 |
| 财务管理员 | 预算、交易、发票、支付、会计导出和支出报表 |
| 采购管理员 | 采购请求、供应商、谈判、合同和续订 |
| IT 管理员 | 应用目录、身份、席位、访问、发现和离职流程 |
| 安全审查员 | 安全问卷、风险、合规证据和审批节点 |
| 审批人 | 查看并处理分配给自己的审批任务 |
| 应用负责人 | 管理被分配应用的业务信息、席位和续订建议 |
| 普通成员 | 提交申请、查看自己的请求、使用分配给自己的应用 |
| 只读审计员 | 查看指定范围数据和审计日志，不可修改 |

### 4.3 权限原则

- 权限由组织、角色、资源范围和操作共同决定。
- 默认拒绝，后端是最终授权边界，前端隐藏按钮不能代替权限检查。
- 组织所有者不可删除自己的最后一个所有者角色。
- 高风险操作需要近期重新认证。
- 平台支持访问必须由客户授权、限定时长并写入审计日志。

## 6. 功能域详细设计

## 6.1 公开官网与 CMS

### 目标

把当前静态复刻页面转为可维护、可索引、可转化的正式公开站。

### 功能

- 使用路径路由替代哈希路由。
- 每个公开页面具备独立标题、描述、Open Graph、Canonical 和结构化数据。
- 支持博客、指南、报告、播客、媒体、客户故事、职位和法律文档内容类型。
- 支持草稿、预览、定时发布、下线、版本记录和编辑审批。
- 支持中文内容，并为后续多语言预留 locale。
- 预约演示、开始使用、联系销售、合作伙伴和媒体联系表单真实提交。
- 表单具备字段校验、反垃圾、同意记录、来源参数和 CRM 同步。
- 定价页支持套餐比较、常见问题和购买/联系入口。
- 自动生成 sitemap、robots、RSS 和内容搜索索引。
- 法律页面保留版本、生效日期和历史版本。

### 验收标准

- 公开链接直接访问返回正确页面，不依赖前端哈希。
- 每个页面在禁用 JavaScript 的 SEO 抓取环境中仍可获得核心元数据。
- 内容编辑无需修改代码即可发布文章或职位。
- 表单重复提交幂等，提交成功后可在管理后台查询。

## 6.2 身份认证与账号安全

### 功能

- 邮箱密码注册和登录。
- 邮箱验证、忘记密码、密码重置。
- 安全 HttpOnly Cookie 会话。
- 会话列表、远程退出和全部设备退出。
- TOTP MFA、恢复码和强制 MFA 策略。
- Google 与 Microsoft OIDC 登录。
- 企业 SAML/OIDC SSO。
- SCIM 用户与群组同步。
- 登录风险控制、速率限制、验证码升级和异常登录通知。
- 账号删除、个人数据导出和营销偏好管理。

### 验收标准

- 密码不进入日志，使用现代密码哈希算法存储。
- 会话令牌可轮换、可撤销并具有明确过期时间。
- 启用组织强制 SSO 后，普通成员不能绕过 SSO。
- MFA 恢复流程不会削弱组织安全策略。

## 6.3 组织、多租户与成员管理

### 功能

- 创建组织、设置名称、地区、默认币种、财年和时区。
- 邀请成员、批量邀请、撤销邀请和重新发送。
- 成员状态：邀请中、活跃、暂停、离职、已删除。
- 自定义角色与权限模板。
- 部门、成本中心、法人实体、地点和项目维度。
- 组织切换和跨组织账号。
- 数据导入、完整导出和组织删除。
- 所有业务表强制包含 `organization_id`。

### 验收标准

- 任意组织成员不能读取或修改其他组织资源。
- 删除组织采用延迟删除和管理员二次确认。
- 成员停用后所有活动会话立即失效。

## 6.4 工作台总览与待办

### 功能

- 软件总支出、应用数量、活跃合同、即将续订、节省机会和风险数量。
- 按部门、成本中心、供应商、类别和时间筛选。
- 我的待办：审批、续订决策、信息补全、安全审查、发票异常。
- 趋势图、预算执行、节省漏斗和数据新鲜度。
- 可配置小组件和组织默认布局。
- 所有数字可下钻到来源记录。

### 验收标准

- 不展示无法追溯到明细的数据。
- 空状态明确提供下一步导入或配置动作。
- 数据过期时显示最后同步时间和失败原因。

## 6.5 应用目录与软件发现

### 功能

- 应用主数据：名称、Logo、类别、供应商、状态、负责人、部门、风险等级。
- 关联席位、用户、合同、交易、发票、采购请求和集成。
- 自定义字段、标签、备注和附件。
- 来源：手工创建、CSV 导入、账单识别、SSO、浏览器、财务系统和企业卡。
- 去重与合并候选，保留合并审计记录。
- 影子 IT 发现、未批准应用和无负责人的应用。
- 应用详情时间线。
- 批量归属、批量标签、批量停用和导出。

### 验收标准

- 同一组织内的供应商别名能归并为统一应用。
- 合并操作可预览影响并可在限定时间内撤销。
- 每条发现记录保留来源、置信度和首次/最后发现时间。

## 6.6 席位、使用率与访问生命周期

### 功能

- 席位数量、已分配、活跃、闲置和超额。
- 用户最后活跃时间、使用频率和数据来源。
- 席位申请、分配、回收和转移。
- 入职模板按角色自动建议应用。
- 离职任务自动生成访问回收清单。
- 无活动阈值和低使用率规则可配置。
- 使用率不足时通知应用负责人。
- 不自动删除账号，除非组织明确启用且集成支持安全撤销。

### 验收标准

- 使用率数据展示采集来源和最后同步时间。
- 自动回收动作默认需要审批。
- 离职流程能证明每个应用的处理状态。

## 6.7 采购申请与审批工作流

### 功能

- 申请类型：新购、增购、升级、续订、试用、紧急采购。
- 动态表单：业务理由、人数、预算、数据类型、集成需求和期望日期。
- 自动查重并推荐现有应用。
- 预算检查、供应商风险、安全问卷、法务与财务节点。
- 串行、并行、任一通过、全部通过和条件分支。
- 金额、部门、数据敏感性、供应商风险和申请类型驱动规则。
- 评论、提及、附件、补充资料和退回修改。
- 委托审批、代理审批、超时提醒和升级。
- 审批决定必须记录理由。
- 通过后可创建应用、供应商、合同草稿和支付请求。

### 状态

`draft -> submitted -> triage -> in_review -> approved/rejected/withdrawn -> fulfilled -> closed`

### 验收标准

- 流程版本在申请提交时冻结，后续修改流程不影响历史申请。
- 同一审批动作使用幂等键，重复点击不会重复执行。
- 所有状态变化均写入不可变事件和审计日志。

## 6.8 供应商与风险管理

### 功能

- 供应商主档、联系人、网站、注册地、税务信息和支付信息引用。
- 安全、隐私、财务和运营风险评分。
- 问卷模板、证据附件、到期日和复审周期。
- 数据处理协议、子处理商、数据区域和合规认证。
- 制裁名单或高风险供应商外部检查通过适配器接入。
- 风险接受、缓解措施、负责人和到期日。
- 供应商事件时间线。

### 验收标准

- 风险评分每个维度可解释，不只展示黑盒总分。
- 过期证据自动降低可信度并触发复审。
- 风险接受需要指定权限和有效期。

## 6.9 合同与续订管理

### 功能

- 合同上传、元数据录入和 AI 字段抽取。
- 合同字段：开始、结束、自动续订、通知期、付款周期、金额、币种、席位和涨价条款。
- 主合同、订单、补充协议和数据协议关联。
- 续订日历和 120/90/60/30/7 天提醒。
- 续订决策：续订、降级、扩容、谈判、取消。
- 谈判任务、报价版本、目标价格和结果。
- 续订责任人、决策截止日和审批。
- 合同版本和附件访问控制。

### 验收标准

- 续订日期由合同字段计算，并可人工覆盖且保留理由。
- 自动续订合同在通知期前触发高优先级提醒。
- 合同解析结果必须经过人工确认后才成为正式字段。

## 6.10 支出、预算与交易

### 功能

- 导入企业卡、银行、会计和 CSV 交易。
- 供应商匹配、应用匹配、类别、部门、成本中心和项目归类。
- 预算年度、期间、维度和版本。
- 实际、承诺、预测和剩余预算。
- 重复扣款、异常涨价、未批准供应商和超预算规则。
- 交易拆分、批量编辑、备注和附件。
- 多币种原币、组织本位币和汇率来源。
- 月结锁定和重开审批。

### 验收标准

- 每笔金额保留原币、汇率、换算时间和换算来源。
- 自动匹配提供置信度，低置信度进入人工队列。
- 锁定期间的数据修改必须由授权用户重开。

## 6.11 支付与虚拟卡

### 功能

- 通过支付供应商适配器创建供应商卡或支付工具。
- 每供应商独立限额、币种、有效期和负责人。
- 即时冻结、解冻、终止和替换。
- 单笔、日、月和总额限制。
- 商户锁定、类别限制和地域限制。
- 支付失败、余额不足、限额触发和可疑交易通知。
- 付款请求与已批准采购关联。
- Webhook 验签、重放保护和事件幂等。

### 安全边界

- 平台不存储完整卡号、CVV 或支付供应商敏感凭证。
- UI 仅展示掩码信息。
- 真实资金能力必须通过具备相应资质的供应商提供。

### 验收标准

- 未批准请求不能创建支付工具。
- 冻结动作在供应商确认前显示处理中，不虚假显示成功。
- Webhook 重试不会重复记账。

## 6.12 发票、收据与会计自动化

### 功能

- 文件上传、邮件转发和 API 导入。
- OCR/AI 抽取供应商、日期、金额、税额、币种、发票号和行项目。
- 发票、交易、合同和应用三方/四方匹配。
- 重复发票检测。
- 科目、税码、成本中心、部门和项目映射。
- 异常队列和人工修正。
- 会计期间导出和同步状态。
- 支持 Xero、QuickBooks、NetSuite 等适配器。

### 验收标准

- 原始文件存储加密，访问使用短期签名 URL。
- OCR 结果必须保留字段级置信度。
- 已同步记录再次修改时进入差异处理流程。

## 6.13 AI 账单审计与退订 Copilot

### 功能

- 输入方式：CSV、PDF、图片、邮件正文、Apple/Stripe/PayPal/Google Play 收据。
- 上传前提示敏感信息边界。
- 文件病毒扫描和格式检查。
- 解析任务异步执行，显示进度、失败原因和重试。
- 识别应用、供应商、金额、币种、周期、日期、来源和证据。
- 风险：疑似闲置、疑似重复、异常涨价、隐藏费用、API 用量、模糊 Apple 扣款和未批准软件。
- 将分析结果匹配到应用目录和交易。
- 人工确认、纠正、忽略和反馈。
- 生成官方取消路径、步骤、支持邮件和取消记录。
- 状态：待确认、已确认活跃、已标记优化、取消中、已取消、取消失败、已验证节省。
- 取消操作由用户执行；平台默认不自动登录第三方网站或代替用户点击取消。
- 对可 API 撤销的企业集成，另走组织管理员启用的受控自动化流程。

### AI 安全

- 输入在调用模型前按策略脱敏。
- 不把客户数据用于模型训练。
- 记录模型、提示版本、处理时间和结果置信度，不记录明文敏感输入到普通日志。
- AI 输出永远是建议，财务和合同字段需要人工确认。

### 验收标准

- 每个识别项展示证据和置信度。
- 用户修正可以反馈到组织别名规则，不自动污染全局规则。
- 节省只有在后续账单确认不再扣款后才记为“已验证”。

## 7. 关键端到端流程

## 7.1 新组织首次使用

1. 用户注册并验证邮箱。
2. 创建组织，填写地区、币种和时区。
3. 邀请成员或跳过。
4. 选择导入方式：账单、应用 CSV、Google/Microsoft 或会计系统。
5. 系统异步解析并生成应用、供应商和待确认记录。
6. 用户确认应用归属和负责人。
7. 总览展示第一批支出、续订和节省机会。
8. 引导配置第一个审批流程或续订提醒。

## 7.2 新软件采购

1. 成员提交需求。
2. 系统检测重复应用和预算。
3. 根据规则分配财务、IT、安全和法务审批。
4. 审批人评论、要求补充或作出决定。
5. 采购管理员记录报价和合同。
6. 通过后创建应用、合同和支付请求。
7. 席位分配给申请人。
8. 全过程进入审计时间线。

## 7.3 续订优化

1. 系统在通知期前生成续订任务。
2. 汇总合同金额、席位、使用率、风险和替代应用。
3. 应用负责人给出续订建议。
4. 财务或采购选择续订、降级、谈判或取消。
5. 需要时进入审批。
6. 记录新合同或取消证据。
7. 后续交易验证节省。

## 7.4 AI 账单审计

1. 用户上传文件或粘贴文本。
2. 系统校验、脱敏、扫描并创建任务。
3. 规则与模型解析结构化项目。
4. 匹配现有应用、供应商和交易。
5. 低置信度进入人工确认。
6. 生成风险和优化机会。
7. 用户执行退订步骤或标记仍在使用。
8. 后续账单验证结果。

## 7.5 员工离职

1. HRIS 同步或管理员标记离职。
2. 系统列出该成员全部应用和权限。
3. 创建回收任务并分配给 IT 或应用负责人。
4. 支持 API 的集成执行受控撤销，其他应用要求人工确认。
5. 回收席位并更新可用数量。
6. 所有动作进入审计记录。

## 8. 系统架构

## 8.1 总体结构

采用模块化单体作为第一阶段生产架构，保持明确领域边界；异步任务独立 Worker。该方案比过早微服务更易交付和保持事务一致性，同时为未来拆分支付、集成和 AI 处理服务保留事件接口。

### 运行组件

- `web`：公开站与企业工作台。
- `api`：FastAPI 领域 API。
- `worker`：异步导入、解析、通知、同步和导出。
- `scheduler`：续订提醒、摘要、周期同步和清理任务。
- `postgres`：正式业务数据。
- `redis`：队列、速率限制、短期缓存和任务状态。
- `object-storage`：发票、合同、证据和导出文件。
- `email-provider`：事务邮件。
- `observability`：日志、指标、追踪和错误监控。

### 前端原则

- React Router 负责路径和权限路由。
- TanStack Query 负责服务端状态、缓存、失效和错误重试。
- Zustand 只保存纯客户端偏好，不复制服务端业务数据。
- React Hook Form + Zod 负责表单与共享校验。
- 设计令牌统一颜色、间距、字体、层级和响应式断点。
- 页面组件只编排，领域逻辑下沉到 hooks、services 和 domain components。
- 所有可修改操作显示加载、成功、失败和冲突状态。

## 10. API 设计

## 10.1 通用约定

- 前缀：`/api/v1`。
- JSON 使用 `snake_case` 与现有 Python 模型一致。

- 错误返回稳定的 `code`、`message`、`field_errors` 和 `request_id`。

## 13. 安全与隐私设计

### 13.1 应用安全

- OWASP ASVS 作为开发检查基线。
- CSRF、XSS、SQL 注入、SSRF、路径穿越和开放重定向防护。
- 严格 CSP、安全响应头和 Cookie 属性。
- API 速率限制和暴力破解保护。
- 依赖漏洞、密钥泄漏和容器扫描。

### 13.2 数据保护

- TLS 传输加密。
- 数据库、对象存储和备份静态加密。
- 集成密钥使用 KMS/秘密管理器包络加密。
- 生产数据访问最小化并审计。
- 非生产环境不复制未经脱敏的生产数据。

### 13.3 隐私

- 数据处理目的和保留期明确。
- 原始账单默认短期保留，组织可配置更短期限。
- 不保存完整卡号和 CVV。
- 不抓取浏览器 Cookie。
- 不自动登录第三方 SaaS。
- 模型供应商必须满足不训练和保留策略。

### 13.4 备份与恢复

- PostgreSQL 自动备份和时间点恢复。
- 对象存储版本和生命周期。
- 定期恢复演练。
- 明确 RPO 15 分钟、RTO 4 小时作为初期目标。

## 14. 错误处理与用户体验

### 14.1 错误分类

- 输入校验错误：指出字段与修复方式。
- 权限错误：不泄露资源是否存在。
- 业务冲突：返回当前状态和可选动作。
- 外部供应商错误：显示处理中或同步失败，不伪造成功。
- 长任务失败：保留已完成阶段和安全重试入口。
- 系统错误：显示请求 ID，不暴露堆栈或密钥。

### 14.2 前端状态

每个数据视图必须包含：

- 初始加载
- 骨架屏
- 空状态
- 权限不足
- 部分数据
- 数据过期
- 可重试错误
- 不可恢复错误
- 成功反馈
- 并发冲突

### 14.3 高风险动作

冻结卡、删除组织、撤销访问、删除集成、导出敏感数据和接受高风险供应商时：

- 显示影响范围。
- 要求明确确认。
- 必要时重新认证。
- 写入审计日志。
- 异步结果必须展示最终状态。

## 20. 明确不做的行为

以下行为不属于默认产品能力：

- 托管客户第三方 SaaS 密码。
- 抓取客户浏览器 Cookie。
- 未经客户明确启用而自动登录第三方网站。
- 未经审批自动取消外部订阅。
- 存储完整卡号或 CVV。
- 把客户数据用于训练公共模型。
- 用 AI 结果自动替代合同、财务或安全的最终决定。
- 在缺少支付资质的情况下自行持有或清算资金。

## 21. 全局完成定义

一个功能只有同时满足以下条件才算完成：

- 产品需求和权限规则已实现。
- API 和 UI 状态完整。
- 数据迁移可执行并可回滚到兼容版本。
- 单元、集成、权限和关键 E2E 测试通过。
- 审计、日志、指标和错误处理已覆盖。
- 可访问性和响应式布局通过验收。
- 文档、运维说明和支持诊断已更新。
- 不包含明文密钥、敏感日志或跨租户访问风险。
- Staging 环境通过验收并由功能开关安全发布。

## 22. 设计决策摘要

1. 采用模块化单体，不在早期拆微服务。
2. 公开站与工作台共享设计系统，但拥有独立路由和布局。
3. PostgreSQL 是业务事实来源，Zustand 不保存正式业务状态。
4. 所有业务实体以组织隔离，后端执行权限。
5. 长任务异步化，所有消费者幂等。
6. 真实支付通过合规供应商 Adapter 接入。
7. AI 提供可解释建议，不替代人工确认。
8. 取消第三方订阅默认由用户执行，不托管第三方凭据。
9. 关键写操作具有不可变审计记录。
10. 先完成应用目录 + 账单审计 + 节省机会的价值闭环，再扩展采购、续订和财务自动化。
