# 企业软件管理平台交付路线图与验收矩阵

## 1. 交付目标

最终产品必须同时具备：

- 可索引、可管理内容、可提交线索的中文公开官网。
- 安全登录、多租户组织和组织级 RBAC。
- 应用目录、发现、席位和访问生命周期。
- AI 账单审计、退订 Copilot 和节省验证。
- 采购审批、供应商风险、合同与续订。
- 预算、交易、支付、发票与会计。
- 集成、报表、安全合规、平台计费和支持后台。
- 可观测性、灾备和生产发布证据。

任何单一页面、API 或演示数据均不能单独证明“全面完成”。

## 3. 发布门禁

### Alpha

- 完成 01-10。
- 单组织试点可登录、导入、确认、发现节省。
- 不启用真实资金操作。
- 关键多租户测试通过。

### Beta

- 完成 11-19。
- 采购、合同、预算、发票和集成可在试点组织使用。
- 支付仅 sandbox 或受控试点。
- 完成首次外部安全测试。

### General Availability

- 完成 20-22。
- 套餐计费、支持、后台、SLO、灾备和安全门禁通过。
- 真实支付供应商资质与合同就绪。
- 全部关键 E2E 在 production-like staging 通过。

## 5. 关键风险与应对

| 风险 | 应对 |
|---|---|
| 当前公开站单体文件过大 | 计划 02 先拆路由、内容和设计系统 |
| 旧 SQLite 数据与新模型不一致 | 计划 04 提供幂等迁移命令，仅迁移结构化记录 |
| 跨租户数据泄露 | Repository 强制组织上下文，计划 06 建立全资源测试 |
| AI 误判影响财务决策 | 计划 09 强制证据、置信度和人工确认 |
| 支付状态与供应商不一致 | 计划 15 使用 pending 状态和 Webhook 最终确认 |
| 集成重复同步 | 计划 17 使用游标、外部 ID 和 Inbox 幂等 |
| 指标口径漂移 | 计划 18 集中 MetricRegistry |
| 支持人员过度访问 | 计划 21 使用时限、范围和客户批准授权 |
| 过早微服务化拖慢交付 | 保持模块化单体，Outbox 为未来拆分准备 |

## 6. 全局 Definition of Done

- 需求、权限、错误状态和审计已实现。
- 单元、集成、权限、E2E 和非功能测试通过。
- 数据迁移在空库和上一版本库执行成功。
- 监控、告警和运行手册可用。
- 可访问性和响应式验收通过。
- 无明文秘密、敏感日志或跨租户漏洞。
- Staging 发布并通过真实流程验收。
- 变更有独立、可回滚或可关闭的提交与功能开关。
